📋 System Overview

What is the Jira Analysis Agent?
A production-ready AI agent built with Spring AI 1.0 that automatically analyzes Jira tickets to provide comprehensive insights including requirements analysis, technical assessment, risk evaluation, and effort estimation.

🎯 Key Features

🔍 Deep Analysis

Comprehensive ticket analysis including functional requirements, technical complexity, and risk assessment.

🔌 Plugin Architecture

Flexible prompt system with hot-reload capabilities and external configuration support.

⚡ High Performance

Caffeine caching, async processing, and optimized file watching for maximum throughput.

📊 Production Ready

Complete observability, health checks, metrics, and enterprise-grade error handling.

🤖 AI Powered

Uses Ollama with LLa MA models for local, private AI processing with Spring AI integration.

🔄 Hot Reload

Update prompts and configurations without restarting the application.

🔄 Data Flow

Request Processing Flow:
  1. API Request → REST Controller receives analysis request
  2. Plugin Loading → Prompt loaded from cache or file system
  3. Jira Integration → Ticket data retrieved via Jira REST API
  4. AI Processing → Spring AI ChatClient processes with Ollama
  5. Tool Execution → File system operations and data storage
  6. Response → Structured JSON analysis returned

📦 Core Components

Component Responsibility Technology
AnalysisController REST API endpoints for ticket analysis Spring Web
OptimizedJiraAnalysisService Core business logic and AI orchestration Spring AI ChatClient
OptimizedPromptPluginManager Plugin loading, caching, and hot-reload Caffeine Cache
JiraRetrievalTool Jira API integration and ticket retrieval WebClient
FileSystemTool Local file operations and storage Java NIO

⚙️ Setup Guide

Prerequisites: Java 21+, Maven 3.8+, Docker (optional), Ollama

Install Ollama

# Install Ollama curl -fsSL https://ollama.ai/install.sh | sh # Pull LLaMA model ollama pull llama3.1:8b # Verify installation ollama list

Clone and Build Project

# Clone repository git clone https://github.com/your-org/jira-analysis-agent.git cd jira-analysis-agent # Build project mvn clean install # Run tests mvn test

Configure Environment

# Create application-local.yml spring: ai: ollama: base-url: http://localhost:11434 chat: options: model: llama3.1:8b # Jira Configuration jira: base-url: https://your-company.atlassian.net username: your-email@company.com api-token: your-jira-api-token # Agent Configuration agent: output-directory: ./analysis-output plugins: external-config-path: ./external-config hot-reload: enabled: true

Set Environment Variables

# Required environment variables export JIRA_BASE_URL=https://your-company.atlassian.net export JIRA_USERNAME=your-email@company.com export JIRA_API_TOKEN=your-jira-api-token # Optional configurations export AGENT_OUTPUT_DIR=./analysis-output export AGENT_EXTERNAL_CONFIG=./external-config export AGENT_HOT_RELOAD=true

Create Plugin Directory Structure

# Create external plugin directories mkdir -p external-config/prompts mkdir -p external-config/agents mkdir -p external-config/skills # Create custom prompt example cat > external-config/prompts/security-analysis-prompt.md << 'EOF' # Security-Focused Jira Analysis Analyze ticket: {{ticketId}} with emphasis on security implications. ## Security Analysis Requirements - Identify potential security vulnerabilities - Assess data privacy implications - Review authentication/authorization requirements - Check for compliance requirements (GDPR, SOX, etc.) Use available tools and provide detailed security recommendations. EOF

Start the Application

# Start with Maven mvn spring-boot:run # Or run the JAR java -jar target/jira-analysis-agent-1.0.0.jar # With custom profile mvn spring-boot:run -Dspring-boot.run.profiles=local

Verify Installation

# Check health curl http://localhost:8080/api/v1/monitoring/health # Check plugin statistics curl http://localhost:8080/api/v1/monitoring/plugin-stats # Test analysis (replace PROJ-123 with real ticket) curl -X POST http://localhost:8080/api/v1/analysis/tickets/PROJ-123

📡 API Reference

🔍 Analysis Endpoints

POST /api/v1/analysis/tickets/{ticketId}

Description: Analyze a Jira ticket using the default analysis prompt.

Parameters:

  • ticketId (path) - Jira ticket ID or key
POST /api/v1/analysis/tickets/{ticketId}/security

Description: Perform security-focused analysis of a Jira ticket.

POST /api/v1/analysis/tickets/{ticketId}/custom

Description: Analyze ticket with custom prompt and context.

{ "promptName": "custom-analysis-prompt", "context": { "projectType": "microservice", "team": "backend" } }

GET /api/v1/monitoring/health

Description: Check agent health status.

GET /api/v1/monitoring/plugin-stats

Description: Get plugin cache statistics.

GET /actuator/health

GET /actuator/metrics

💡 Usage Examples

🎯 Example 1: Basic Analysis

# Analyze a feature request ticket curl -X POST http://localhost:8080/api/v1/analysis/tickets/FEAT-456 \ -H "Content-Type: application/json" \ | jq '.requirements_analysis.functional_requirements' # Output: List of functional requirements [ "User registration with email verification", "Profile management interface", "Password reset functionality" ]

🔒 Example 2: Security Analysis

# Security-focused analysis curl -X POST http://localhost:8080/api/v1/analysis/tickets/SEC-789/security \ -H "Content-Type: application/json" \ | jq '.risk_assessment.identified_risks[] | select(.category == "SECURITY")' # Output: Security-specific risks { "description": "Potential SQL injection vulnerability", "category": "SECURITY", "impact": "HIGH", "probability": "MEDIUM", "mitigation_strategy": "Use parameterized queries and input validation" }

⚡ Example 3: Performance Analysis

# Create custom performance prompt cat > external-config/prompts/performance-prompt.md << 'EOF' # Performance Analysis for {{ticketId}} Focus on performance implications: - Database query optimization - Caching strategies - Scalability concerns - Load testing requirements Expected load: {{context.expectedLoad}} Current baseline: {{context.baseline}} EOF # Use custom prompt with context curl -X POST http://localhost:8080/api/v1/analysis/tickets/PERF-101/custom \ -H "Content-Type: application/json" \ -d '{ "promptName": "performance-prompt", "context": { "expectedLoad": "10000 concurrent users", "baseline": "500ms response time" } }'

🔄 Example 4: Hot Reload Demo

# 1. Create new prompt echo "# Quick Analysis for {{ticketId}}" > external-config/prompts/quick-prompt.md echo "Provide a brief 3-sentence analysis." >> external-config/prompts/quick-prompt.md # 2. Use immediately (no restart needed!) curl -X POST http://localhost:8080/api/v1/analysis/tickets/TEST-123/custom \ -d '{"promptName": "quick-prompt"}' # 3. Update prompt echo "# Updated Quick Analysis for {{ticketId}}" > external-config/prompts/quick-prompt.md echo "Provide analysis with focus on risks and effort." >> external-config/prompts/quick-prompt.md # 4. Use updated version (automatically reloaded!) curl -X POST http://localhost:8080/api/v1/analysis/tickets/TEST-123/custom \ -d '{"promptName": "quick-prompt"}'
