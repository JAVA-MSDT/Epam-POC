3. File 02 — System Architecture & Advanced Concepts

File 02

Architecture Introduced

File 02 introduced the first SVG workflow diagram showing the Orchestrator at the top, dispatching to 8 agents (split left/right), with Jira and GitHub as external systems. A second, simpler architecture SVG showed: Orchestrator → Agent(s) → Claude LLM API (triangle).

Advanced Agent Concepts

  • Stateless vs. Stateful: Stateless agents process each request independently (recommended for scaling). Stateful agents maintain conversation/workflow history.
  • Single Responsibility Principle (SRP): Each agent = one workflow step. Makes agents composable and testable.
  • Agent Communication: Via orchestrator (recommended for control) or peer-to-peer (event-driven).
  • Security: Least-privilege API keys; log all actions for auditability.
  • Error Handling: Retries, fallbacks, and human escalation for robustness.
  • Observability: Monitor performance, collect metrics, log errors.
  • Extensibility: Agents as plug-ins — add/swap without breaking the workflow.

4. File 03 — Optimization & Human-in-the-Loop

File 03

Optimization Analysis

File 03 is the critical optimization layer. It analyzed which of the 8 steps could be combined, pipelined, or must remain separate, and where humans must intervene.

Steps Strategy Human Checkpoint? Reason
1 + 2 Combined (one LLM call) Yes — after Human approves project structure before proceeding
3 + 4 Pipeline (sequential) No Deep Dive feeds directly into Visual Report — automated
5 Separate Yes — during Human reviews clarifications, provides feedback
6 + 7 Pipeline (sequential) No QA validates generated code — fully automated
8 Separate Yes — before/after Human confirms deployment — critical safety gate
Key efficiency gain: From 8 separate LLM calls down to 5 optimized calls (1+2 combined, 3+4 piped, 6+7 piped), with 3 human gates for quality control.

Optimized Orchestrator Pseudocode (from File 03)

// Step 1+2: one combined LLM call var result12 = agents.ticketAndSetup(ctx.ticketText); if (!humanConfirm("Ticket & Setup", ctx)) return; // Step 3+4: pipeline var result34 = agents.deepDiveAndVisual(ctx.ticketSummary, ctx.projectSetup); // Step 5: separate + human gate ctx.reviewNotes = agents.review(ctx.deepDive, ctx.visualReport); if (!humanConfirm("Review & Clarification", ctx)) return; // Step 6+7: pipeline var result67 = agents.implementationAndQA(ctx.reviewNotes); // Step 8: separate + human gate ctx.deploymentStatus = agents.deployment(ctx.implementation, ctx.qaReport); humanConfirm("Deployment & Review", ctx);

7. Distilled Key Findings

Synthesizing all five files, the following non-obvious insights stand out:

Finding 1 — Markdown is the Agent Contract: The markdown files in resources/agents/ are not documentation — they are the actual LLM prompts. Every agent class has exactly one corresponding .md file. Changing agent behavior means editing markdown, not Java.
Finding 2 — 5 LLM Calls, Not 8: File 03 proved that the naive 1-step=1-call design wastes tokens and latency. The optimized design combines steps 1+2, pipelines 3→4 and 6→7, reducing to 5 LLM calls total while preserving all workflow outputs.
Finding 3 — WorkflowContext is the Backbone: All agents share a single WorkflowContext object. This is the single source of truth — it carries data from step 1 all the way to step 8. Agents read from and write to it; they do not call each other directly.
Finding 4 — Three Human Gates: Human confirmation is mandatory after Steps 2, 5, and 8. This is not optional — it is the safety architecture. The humanConfirm() method returns false to halt the workflow.
Finding 5 — Spring Migration is Free: Because agents are stateless, the orchestrator is a plain class, and LLM clients are injected via interface, the entire project converts to Spring Boot by adding @Component / @Service annotations — no structural refactoring needed.

9. Implementation Plan — Phase by Phase

This plan translates all five HTML documents into a concrete build sequence. Each phase is self-contained and produces testable output.

1Maven Project Scaffolding

Initialize the project using Maven archetype, then create all package directories.

mvn archetype:generate \ -DgroupId=com.javamsdt.aidevworkflow \ -DartifactId=aidevworkflow \ -DarchetypeArtifactId=maven-archetype-quickstart \ -DinteractiveMode=false

Add to pom.xml: OkHttp (or Apache HttpClient) for HTTP, Jackson (or Gson) for JSON parsing, JUnit 5 for testing.

Output: Compilable Maven project with all packages present.

2LLM Abstraction Layer

Create the pluggable LLM interface and two concrete implementations.

// LlmClient.java public interface LlmClient { String completePrompt(String prompt); } // ClaudeApiClient.java — calls Anthropic Messages API // OpenAiApiClient.java — calls OpenAI Chat Completions API

Output: Agents can be written against LlmClient without knowing which LLM is active. Swap LLMs via constructor injection.

3WorkflowContext — Shared State

Create the WorkflowContext POJO. Every agent reads from and writes to this object. It travels through all 8 steps and accumulates results.

public class WorkflowContext { public String ticketText; public String ticketSummary; public String projectSetup; public String deepDive; public String visualReport; public String reviewNotes; public String implementation; public String qaReport; public String deploymentStatus; }

Output: Single source of truth that eliminates inter-agent direct calls.

4MarkdownLoader Utility

Utility class that loads a markdown agent file from the classpath and performs {{variable}} substitution.

public class MarkdownLoader { public static String load(String resourcePath) { // Load from classpath: src/main/resources/ InputStream is = MarkdownLoader.class .getClassLoader() .getResourceAsStream(resourcePath); return new String(is.readAllBytes(), StandardCharsets.UTF_8); } }

Output: Agents load prompts with one line: MarkdownLoader.load("agents/ticket_analysis.md")

68 Agent Java Classes

Output: 8 stateless, independently testable agent classes.

7WorkflowOrchestrator — Optimized 5-Call Flow

Implement the orchestrator with combined/pipelined steps from File 03's optimization. Insert the three human confirmation gates.

public class WorkflowOrchestrator { private final LlmClient llmClient; private final WorkflowContext ctx; // Phase 1+2: combined call (1 LLM call) public boolean runTicketAndSetup() { ... } // Phase 3+4: pipeline (1 LLM call → feeds next) public void runDeepDiveAndVisual() { ... } // Phase 5: review + human gate (1 LLM call + human) public boolean runReview() { ... } // Phase 6+7: pipeline (1 LLM call → validates self) public void runImplementationAndQA() { ... } // Phase 8: deployment + human gate (1 LLM call + human) public boolean runDeployment() { ... } // Human gate — CLI stub, replaceable with UI/Slack/email private boolean humanConfirm(String step) { System.out.println("[HUMAN GATE] Approve step: " + step + "? (y/n)"); return new Scanner(System.in).nextLine().trim().equalsIgnoreCase("y"); } }

Output: Full workflow in 5 LLM calls. Three humanConfirm() gates halt on rejection.

8Unit Tests

Test each layer independently using mock LlmClient implementations. Use JUnit 5 + Mockito.

  • MarkdownLoaderTest — verifies template loading and variable substitution.
  • TicketAnalysisAgentTest (and all other agents) — mock LlmClient, assert context writes.
  • WorkflowOrchestratorTest — mock agents, assert workflow sequencing and human gates.
  • ClaudeApiClientTest — integration test (optional, needs API key).

Output: Test suite that passes with zero real LLM calls.

9Spring Boot Migration Path (Optional Phase)

When ready, convert to Spring Boot by:

  • Add spring-boot-starter to pom.xml.
  • Annotate each agent class with @Component.
  • Annotate WorkflowOrchestrator with @Service.
  • Annotate LlmClient implementations with @Bean / @Profile.

10. Architecture Decision Record

Decision 1 — Combine Steps 1 + 2

Ticket Analysis + Project Setup as one LLM call

Rationale: Both steps share the same input (ticket text) and the output of step 1 (ticket summary) is immediately consumed by step 2 (project setup). Combining them into one prompt reduces latency and token overhead by eliminating a round-trip.

Trade-off: Combined prompt is slightly more complex to write. Mitigated by structured markdown output format (## Summary / ## Project Setup sections).

Decision 2 — Pipeline Steps 3 + 4

Deep Dive output feeds directly into Visual Report

Rationale: The Visual Report (step 4) has no independent input — it consumes the Deep Dive analysis (step 3). Pipelining them keeps the LLM "warm" on the same context and avoids redundant context re-feeding.

Trade-off: Visual Report cannot run without Deep Dive completing first. This is acceptable since data dependency already enforces sequencing.

Decision 3 — Pipeline Steps 6 + 7

QA validates the code generated by Implementation in the same call

Rationale: QA (step 7) needs the implementation (step 6) output immediately. Running them in sequence with context sharing lets the QA prompt include the generated code inline, producing a self-contained analysis.

Trade-off: If implementation generates a very large code block, the combined prompt may approach context limits. Mitigated by limiting implementation output to code outlines and key snippets, not full file dumps.

Decision 4 — Three Human Gates, Not More

Human checkpoints after steps 2, 5, and 8 only

Rationale: Over-checking with humans defeats the automation purpose. The three gates cover the three irreversible transitions: (1) committing to a project structure, (2) approving the analysis before writing code, (3) authorizing deployment. All other steps are safe to automate.

Decision 5 — Interface-Driven LlmClient

Agents depend on LlmClient interface, never on Claude or OpenAI directly

Rationale: LLM providers change APIs, pricing, and capabilities. Isolating all agents from the concrete LLM means switching providers is a one-line change in the orchestrator's constructor (or a Spring @Profile).

Decision 6 — Markdown as the Agent Contract

Agent behavior defined in .md files, not in Java strings

Rationale: Non-engineers (product managers, domain experts) can refine agent behavior by editing markdown files without touching Java. This separates concerns cleanly: Java handles orchestration logic; markdown handles LLM instruction.
