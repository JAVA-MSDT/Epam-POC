2. Finding — BalanceMaskingCondition Design

Critical

This is the core new class. It must follow the exact pattern of PhoneMaskingCondition (the only existing custom Spring-managed condition), but with fundamentally different comparison logic.

F1.1 — Input type mismatch: setInput receives Object, but threshold is a numeric string

The MaskMeCondition.setInput(Object) contract passes the raw header value as a String. The condition must parse it to BigDecimal internally. If the header contains "abc" or is empty, a NumberFormatException will crash the masking pipeline.

🔧 Suggested Fix Parse inside setInput with try-catch. On failure, set threshold to null so shouldMask returns false (no masking when input is invalid).
⚡ Impact If not handled: any invalid header value will throw an unhandled exception, returning a 500 to the client on ALL endpoints that use this condition.
🧪 Edge Cases
  • Header value: "" (empty string) — must not mask
  • Header value: "abc" — must not mask, must not throw
  • Header value: "-10" — negative threshold, all positive balances exceed it
  • Header value: "0" — any non-zero balance exceeds it
  • Header value: "100.999999999" — high precision BigDecimal
  • Header absent entirely — setInput never called, threshold stays null
F1.2 — shouldMask needs access to the actual BigDecimal field value, but receives Object

shouldMask(Object maskedFieldValue, Object objectContainingMaskedField) — the maskedFieldValue is the raw field value. For balance it will be a BigDecimal. You must cast safely.

🔧 Suggested Fix Check maskedFieldValue instanceof BigDecimal before comparing. If it's not a BigDecimal (e.g., the annotation is accidentally placed on a String field), return false.
🧪 Edge Cases
  • maskedFieldValue is null — user has null balance → must not mask, must not NPE
  • maskedFieldValue is BigDecimal.ZERO — threshold "0" with strictly-greater-than → should NOT mask
F1.3 — Strictly greater than vs. greater-than-or-equal (Open Question #1)

The requirements say "exceeds a threshold". The word "exceeds" in English means strictly greater than. However, the open question asks for clarification.

🔧 Recommendation Use balance.compareTo(threshold) > 0 (strictly greater than). This matches the natural meaning of "exceeds" and is the safer default — a balance exactly equal to the threshold remains visible.
🧪 Edge Case Balance = 50.00, Threshold = 50.00 → with > it stays visible; with >= it gets masked. Document the chosen behavior.
F1.4 — Spring bean scope: Singleton condition with mutable state

Looking at PhoneMaskingCondition, it stores private String input as instance state. Since it's a @Component (singleton by default), the input field is shared across all requests. The MaskMe library calls setInput before shouldMask on the same thread, but in a concurrent server, two requests could interleave.

However, examining MaskMeProcessor, it uses MaskMeConditionFactory.createCondition() which calls applicationContext.getBean(type). Since Spring returns the singleton, the setInput → shouldMask sequence is NOT atomic across threads.

🔧 Suggested Fix This is an existing pattern issue (also affects PhoneMaskingCondition). For your new condition, follow the same pattern for consistency, but be aware this is a latent concurrency bug in the demo app. For production, consider @Scope("prototype") or ThreadLocal for the input field.
⚡ Impact on existing code This is NOT a new risk — it already exists in PhoneMaskingCondition. Following the same pattern keeps the codebase consistent.

Pattern to follow — based on PhoneMaskingCondition.java:

@Component
@RequiredArgsConstructor
public class BalanceMaskingCondition implements MaskMeCondition {
    private BigDecimal threshold;  // set via setInput

    @Override
    public boolean shouldMask(Object maskedFieldValue, Object objectContainingMaskedField) {
        if (threshold == null || !(maskedFieldValue instanceof BigDecimal balance)) {
            return false;
        }
        return balance.compareTo(threshold) > 0;
    }

    @Override
    public void setInput(Object input) {
        if (input instanceof String s && !s.isBlank()) {
            try { this.threshold = new BigDecimal(s); }
            catch (NumberFormatException e) { this.threshold = null; }
        }
    }
}

3. Finding — CustomBigDecimalConverter

High

The requirement says "a custom converter to handle BigDecimal type conversion". But the MaskMe library already has MaskMeNumberConverter which handles BigDecimal — it converts empty strings to BigDecimal.ZERO. So why do we need a custom one?

F2.1 — Potential conflict with built-in MaskMeNumberConverter

The built-in MaskMeNumberConverter (priority 0) already handles BigDecimal. If you register a CustomBigDecimalConverter with priority > 0, it will intercept ALL BigDecimal conversions globally — not just for the balance field.

This means the balance field on the User domain entity (which still uses AlwaysMaskMeCondition) will also be affected by the new converter.

🔧 Suggested Fix
  • Option A: Don't create a custom converter at all — the built-in MaskMeNumberConverter already converts empty maskValue to BigDecimal.ZERO. Only create one if you need custom masked display (e.g., "-1" or "[HIDDEN]").
  • Option B: If you do create one, use the maskedFieldName parameter to scope it: only apply custom logic when fieldName.equals("balance"), and return null for other fields to let the chain continue.
⚡ Impact A global BigDecimal converter with priority > 0 will affect: UserDto.balance, User.balance, and any future BigDecimal field in any DTO/entity. This is a silent behavioral change to existing endpoints.
F2.2 — canConvert scope: BigDecimal only or all Number types?

Following the pattern of CustomStringConverter which returns type == String.class, the new converter should return type == BigDecimal.class — NOT all numeric types.

🔧 Suggested Fix canConvert must check type == BigDecimal.class only. If you check Number.class.isAssignableFrom(type), you'll intercept Integer, Long, Double conversions too — breaking GeoLocationDto.latitude (Double) and UserDto.id (Long).
F2.3 — Priority value must be > 0 to override built-in, but consider side effects

The existing CustomStringConverter uses priority 10. The built-in MaskMeNumberConverter uses priority 0. Your converter needs priority > 0 to take precedence.

🔧 Recommendation Use priority 10 to match the existing custom converter pattern. The converter chain in MaskMeConverterRegistry sorts by priority descending, so higher = tried first.

Pattern to follow — based on CustomStringConverter.java:

public class CustomBigDecimalConverter implements MaskMeConverter {
    @Override
    public int priority() { return 10; }

    @Override
    public boolean canConvert(Class<?> type) {
        return type == BigDecimal.class;
    }

    @Override
    public Object convert(String maskingValue, Class<?> targetFieldType,
                          Object originalFieldValue, Object containingObject,
                          String maskedFieldName) {
        if (maskingValue == null || maskingValue.isBlank()) {
            return BigDecimal.ZERO;
        }
        try { return new BigDecimal(maskingValue); }
        catch (NumberFormatException e) { return BigDecimal.ZERO; }
    }
}

4. Finding — UserDto Annotation Change

Critical
F3.1 — Changing balance condition BREAKS existing endpoints

Currently UserDto.balance uses AlwaysMaskMeCondition. The requirement says to replace it with BalanceMaskingCondition. This means:

  • GET /users/masked/{id} — currently masks balance unconditionally → will now only mask when threshold header is provided AND balance exceeds it
  • GET /users — currently masks balance unconditionally → same behavioral change
  • GET /users/user/{id} — uses User domain entity, NOT UserDto → unaffected
⚡ Impact — This is the highest-risk change
  • Acceptance criteria says "All existing endpoints remain unchanged and functional" — but changing the annotation on UserDto.balance directly changes the behavior of GET /users/masked/{id} and GET /users
  • Previously: balance was ALWAYS masked on these endpoints
  • After change: balance is only masked when Mask-Balance-Threshold header is present and balance exceeds it
🔧 Suggested Fix — Two options
  • Option A (Recommended): Use BOTH conditions: @MaskMe(conditions = {AlwaysMaskMeCondition.class, BalanceMaskingCondition.class}). The MaskMe library evaluates conditions with OR logic (any condition returning true triggers masking). This preserves existing behavior AND adds threshold-based masking.
  • Option B: Replace with only BalanceMaskingCondition as the requirement states, but acknowledge this is a breaking change to existing endpoints. Update acceptance criteria accordingly.
🧪 Verification After implementation, test: curl -H "Mask-Input: maskMe" http://localhost:9090/users/masked/1 — balance should still be masked if Option A is chosen.
F3.2 — maskValue for balance: what should the masked display be?

Currently maskValue = "" which the built-in converter turns into BigDecimal.ZERO. The requirement's open question asks: zero, "-1", or something else?

🔧 Recommendation Keep maskValue = "" → BigDecimal.ZERO. This is the current behavior and is semantically clear ("balance hidden = 0"). Using "-1" could confuse API consumers into thinking the user has negative balance.

Current vs. Proposed (Option A):

// CURRENT
@MaskMe(conditions = {AlwaysMaskMeCondition.class}, maskValue = "")
BigDecimal balance

// PROPOSED (Option A — preserves backward compatibility)
@MaskMe(conditions = {AlwaysMaskMeCondition.class, BalanceMaskingCondition.class}, maskValue = "")
BigDecimal balance

5. Finding — MaskMeConfiguration Registration

Medium
F4.1 — BalanceMaskingCondition does NOT need a @Bean declaration

Unlike AlwaysMaskMeCondition and MaskMeOnInput (which are library classes and need explicit @Bean declarations), BalanceMaskingCondition is YOUR class with @Component. Spring's component scanning will find it automatically.

🔧 Suggested Fix Do NOT add a @Bean method for BalanceMaskingCondition in MaskMeConfiguration. The @Component annotation is sufficient. Adding both would create a duplicate bean conflict.
⚡ Impact if done wrong Spring will throw NoUniqueBeanDefinitionException at startup if both @Component and @Bean exist for the same type.
F4.2 — CustomBigDecimalConverter MUST be registered in setupCustomConverters()

The converter is NOT a Spring bean — it's registered via MaskMeConverterRegistry.registerGlobal(). It must be added in the setupCustomConverters() method alongside CustomStringConverter.

🔧 Exact change needed in MaskMeConfiguration.java 
private void setupCustomConverters() {
    MaskMeConverterRegistry.clearGlobal();
    MaskMeConverterRegistry.registerGlobal(new CustomStringConverter());
    MaskMeConverterRegistry.registerGlobal(new CustomBigDecimalConverter()); // ADD THIS
}
⚡ Impact If forgotten: the built-in MaskMeNumberConverter handles BigDecimal already (converts empty to ZERO). So the app won't crash — but any custom conversion logic you intended won't execute.

6. Finding — New Endpoint & Controller Changes

High
F5.1 — New endpoint must follow existing MaskMeInitializer.mask() pattern

Looking at the existing controller, every masked endpoint calls MaskMeInitializer.mask(dto, ConditionClass.class, inputValue). The new endpoint must follow this exact pattern with BalanceMaskingCondition.class and the header value.

🔧 Suggested Implementation 
@GetMapping("/masked/balance/{id}")
public UserDto getMaskedByBalance(@PathVariable Long id,
        @RequestHeader(value = "Mask-Balance-Threshold", defaultValue = "") String threshold) {
    UserDto userDto = userMapper.toDto(userService.findUserById(id));
    return MaskMeInitializer.mask(userDto, BalanceMaskingCondition.class, threshold);
}
🧪 Edge Cases
  • Header absent → defaultValue = "" → setInput("") → threshold is null → no masking ✓
  • Header = "50", user balance = 20.0 → no masking ✓
  • Header = "50", user balance = 100.46 → masked ✓
  • Invalid user ID → existing RuntimeException("User not found") propagates as 500 — consider adding proper error handling
F5.2 — Open Question #3: Should the new endpoint also mask other fields?

If the endpoint only passes BalanceMaskingCondition.class as input, then MaskMeOnInput fields (name, zipCode) will NOT be masked (no input provided for that condition). However, AlwaysMaskMeCondition fields (id, email, password, birthDate, createdAt, city, geoLocation) WILL still be masked because AlwaysMaskMeCondition.shouldMask() always returns true.

🔧 Recommendation This is correct behavior — the endpoint demonstrates balance-threshold masking specifically. AlwaysMask fields mask regardless. MaskMeOnInput fields stay unmasked unless you also pass that condition. This is a feature, not a bug.
F5.3 — URL path design: /users/masked/balance/{id} vs /users/balance/{id}

The sprint board ticket (MASK-102) references GET /users/masked/balance/{id}. This is consistent with the existing /users/masked/{id} pattern. Keep this path.

⚡ No conflict Spring MVC can distinguish /users/masked/{id} from /users/masked/balance/{id} because "balance" is a literal segment, not a path variable.

7. Finding — GET /users Endpoint Integration

Medium
F6.1 — Open Question #4: Should GET /users support balance threshold?

The existing GET /users endpoint already combines two conditions via getMaskedConditionsForGetAllUsers() which returns Object[]{MaskMeOnInput.class, maskInput, PhoneMaskingCondition.class, maskPhone}.

Adding a third condition means extending this array to 6 elements and adding a third @RequestHeader parameter.

🔧 Suggested Fix — If yes, add to GET /users 
@GetMapping
public List<UserDto> getUsers(
        @RequestHeader(value = "Mask-Input", defaultValue = "maskMe") String maskInput,
        @RequestHeader(value = "Mask-Phone", defaultValue = "") String maskPhone,
        @RequestHeader(value = "Mask-Balance-Threshold", defaultValue = "") String balanceThreshold) {

    return userService.findUsers().stream()
            .map(user -> MaskMeInitializer.mask(userMapper.toDto(user),
                    MaskMeOnInput.class, maskInput,
                    PhoneMaskingCondition.class, maskPhone,
                    BalanceMaskingCondition.class, balanceThreshold))
            .toList();
}
⚡ Impact
  • The defaultValue = "" ensures backward compatibility — if the header is absent, BalanceMaskingCondition receives empty string → threshold is null → no masking
  • Existing callers sending only Mask-Input and Mask-Phone headers will see NO behavioral change
  • The getMaskedConditionsForGetAllUsers helper method must be updated to accept and include the third condition pair

F11.1 — BigDecimal scale comparison subtlety

new BigDecimal("100.46").compareTo(new BigDecimal("100.46")) returns 0 ✓. But new BigDecimal("100.46").equals(new BigDecimal("100.460")) returns false because equals checks scale. Always use compareTo, never equals for BigDecimal comparison.

🔧 Already handled The suggested implementation uses compareTo. Just don't accidentally switch to equals.

F12.1 — Converter registration order matters

In setupCustomConverters(), clearGlobal() is called first, then converters are registered. If CustomBigDecimalConverter is registered BEFORE CustomStringConverter, it doesn't matter because they handle different types. But if someone later creates a converter that handles both, registration order + priority determines behavior.

🔧 Recommendation Register in logical order: CustomStringConverter first (existing), then CustomBigDecimalConverter (new). Both use priority 10, but they target different types so no conflict.

14. Finding — Thread Safety & Concurrency

Medium
F13.1 — Singleton condition with mutable threshold field (detailed analysis)

Tracing the full call chain for concurrent requests:

  1. Request A arrives: MaskMeInitializer.mask(dto, BalanceMaskingCondition.class, "50")
  2. Inside mask(): processor.setConditionInput(BalanceMaskingCondition.class, "50") — stored in ThreadLocal ✓
  3. Inside processor.process() → shouldMask() → MaskMeConditionFactory.createCondition(BalanceMaskingCondition.class)
  4. Factory calls applicationContext.getBean(BalanceMaskingCondition.class) → returns SINGLETON
  5. condition.setInput("50") — sets this.threshold = 50 on the SHARED singleton
  6. Request B arrives simultaneously: condition.setInput("200") — overwrites to 200
  7. Request A's shouldMask() now uses threshold=200 instead of 50 ❌
🔧 Mitigation Options
  • For this demo app: Accept the risk — it's a demo, not production. Follow the same pattern as PhoneMaskingCondition for consistency.
  • For production: Use @Scope("prototype") on the condition so each getBean() call returns a new instance. Or use ThreadLocal<BigDecimal> for the threshold field.
⚡ Impact This is a pre-existing architectural pattern in the demo. Changing it for BalanceMaskingCondition alone would be inconsistent. Flag it but don't fix it in this ticket.
