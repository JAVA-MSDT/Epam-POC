📗 Story MASK-101

Add Balance Masking Condition with Threshold Input

Description

Currently the balance field in UserDto uses AlwaysMaskMeCondition, which masks it unconditionally on every request. We need a new BalanceMaskingCondition that masks the balance only when the user's balance exceeds a threshold value provided at runtime via the Mask-Balance-Threshold HTTP header.

This requires a new condition class, a custom BigDecimalConverter, updating the @MaskMe annotation on the balance field, registering the condition as a Spring bean, and exposing a new endpoint.

Requirements
  • A new custom masking condition that masks the balance field only when its value exceeds a runtime threshold
  • The threshold value should be provided via a Mask-Balance-Threshold HTTP header
  • A custom converter to handle BigDecimal type conversion during masking (the existing converter only handles String fields)
  • The new condition must be Spring-managed to support dependency injection
  • The balance field in UserDto should use the new condition instead of AlwaysMaskMeCondition
  • A new REST endpoint to demonstrate the threshold-based balance masking
  • The new condition and converter must be registered in the existing MaskMe configuration
Acceptance Criteria
  • When the threshold header is provided and the user's balance exceeds it, the balance is masked
  • When the threshold header is provided and the user's balance is below it, the balance remains visible
  • When no threshold header is provided, the balance is not masked
  • The new condition supports Spring dependency injection
  • Empty mask values are converted to BigDecimal.ZERO
  • All existing endpoints remain unchanged and functional
⚠️ Open Questions / Clarifications Needed
  • Should the threshold comparison be strictly greater than or greater than or equal?
  • What should the masked balance display as — zero, a placeholder like "-1", or something else?
  • Should the new endpoint also mask other fields (e.g. via MaskMeOnInput), or only the balance?
  • Should the GET /users endpoint support the balance threshold as an additional condition?

📗 Story MASK-103

Add Viewer-Based Visibility Masking for Email Field

Description

Currently the email field in UserDto uses AlwaysMaskMeCondition, which masks it on every request regardless of who is viewing. We need a new VisibilityMaskingCondition that masks the email based on the viewer's relationship to the profile owner.

The visibility level is provided at runtime via a Mask-Visibility HTTP header with three possible values: owner, friend, or public. Only public viewers should see the email masked.

Note: In a production application, the visibility level would be determined by querying user preferences and relationships from a database (e.g., a UserPreferencesRepository). For this demo, we simplify by passing the visibility level directly via an HTTP header.

Requirements
  • A new VisibilityMaskingCondition implementing MaskMeCondition
  • The condition receives the viewer's visibility level via setInput(Object)
  • Three visibility levels: owner (no masking), friend (no masking), public (masked)
  • The visibility level is provided via a Mask-Visibility HTTP header
  • The condition must be Spring-managed (@Component) to support dependency injection
  • The email field in UserDto should use the new condition instead of AlwaysMaskMeCondition
  • A new REST endpoint GET /users/profile/{id} to demonstrate viewer-based masking
  • The new condition must be registered via the Spring framework provider (no @Bean needed — @Component is sufficient)
  • When no header is provided, default to public (most restrictive — mask by default)
Acceptance Criteria
  • When Mask-Visibility: public — email is masked
  • When Mask-Visibility: friend — email is visible
  • When Mask-Visibility: owner — email is visible
  • When no header is provided — email is masked (defaults to public)
  • Invalid header values (e.g., "admin", "") — email is masked (treated as public)
  • The condition supports Spring dependency injection
  • All existing endpoints remain unchanged and functional
  • The GET /users endpoint optionally supports the visibility header as an additional condition
⚠️ Open Questions / Clarifications Needed
  • Should the visibility condition apply only to email, or also to other fields like phone and balance?
  • Should friend visibility partially mask the email (e.g., a***@mail.com) or show it fully?
  • In production, the visibility would come from a DB call — should the condition accept a UserPreferencesService via DI as a placeholder for future integration?
  • Should the new endpoint also trigger AlwaysMaskMeCondition fields (password, city), or only the visibility-based fields?
